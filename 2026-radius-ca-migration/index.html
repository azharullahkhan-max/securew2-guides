DigiCert G1 → G3 Trust Migration
for RADIUS Authentication

DigiCert is deprecating its G1 root CA hierarchy. This guide walks your team through updating trusted CA certificates across every managed platform to maintain uninterrupted RADIUS-based Wi-Fi and VPN authentication.

iOS macOS Windows Android Cisco Meraki VMware Workspace ONE Okta Device Trust

! What's Changing

Understanding the DigiCert root CA transition

DigiCert is retiring its G1 (SHA-1 / legacy RSA) root certificate hierarchy in favour of the modern G3 (SHA-256 / RSA 4096) hierarchy. Any RADIUS server certificate currently chaining to a DigiCert G1 root will be re-issued under G3.

🔴
Authentication will break if you don't act Devices that only trust the G1 root will reject the new G3-signed server certificate and fail 802.1X authentication after the cutover date.
ℹ️
No user credentials change This migration only updates which CA certificates devices trust. Usernames, passwords, and certificates issued to devices or users are not affected.

📱 iOS

iPhone & iPad — manual profile install and Jamf Pro deployment

Manual Installation Manual

Prerequisites

  • iOS 14 or later
  • G3 root CA certificate file (.cer or .mobileconfig) from SecureW2 portal
  • Unsupervised devices: access to Settings → General → VPN & Device Management
  • Supervised devices: the MDM-pushed profile covers trust automatically — no manual profile install required
✅
Supervised devices can skip steps 1–4 If your devices are Apple-supervised and managed by an MDM, the trust profile will be installed silently via MDM push. Confirm by checking Settings → General → VPN & Device Management for an existing SecureW2 profile before proceeding.
1
Download the G3 trust profile

Log in to the SecureW2 JoinNow portal and navigate to Resources → CA Certificates → DigiCert G3 Root. Download the .mobileconfig file to the device.

2
Open the profile

Tap the downloaded file. iOS will show a "Profile Downloaded" banner. Go to Settings → General → VPN & Device Management and tap the profile under Downloaded Profile.

3
Install and trust the certificate

Tap Install, enter your device passcode, then tap Install again on the warning screen. Tap Done.

4
Enable full trust (if prompted)

Go to Settings → General → About → Certificate Trust Settings. Toggle on the DigiCert G3 root under Enable Full Trust For Root Certificates.

5
Remove the G1 profile

In Settings → General → VPN & Device Management, find the old DigiCert G1 profile and tap Remove Profile.

6
Reconnect to Wi-Fi

Toggle Wi-Fi off and on. The device should authenticate successfully using the new G3-signed RADIUS certificate.

✅
Confirmation: The device joins the 802.1X network without a certificate trust prompt and maintains a stable connection.

Jamf Pro Deployment MDM

Prerequisites

  • Jamf Pro 10.40 or later
  • Devices enrolled via Automated Device Enrollment (ADE) or User-initiated Enrollment
  • DigiCert G3 root CA .cer file uploaded to Jamf Pro
  • Admin access to Jamf Pro console
1
Upload the G3 certificate

In Jamf Pro go to Settings → Global Management → PKI Certificates. Click Upload and select the DigiCert G3 root .cer file. Name it clearly, e.g. DigiCert-G3-Root.

2
Create a Configuration Profile

Navigate to Devices → Configuration Profiles → New. Add a Certificate payload. Select the uploaded G3 cert and set Allow apps access to Yes.

3
Set scope

Under the Scope tab add the target device groups or all managed iOS devices. Exclude any test group you use for staged rollouts.

4
Deploy and verify

Click Save. Jamf will push the profile at the next check-in (or force a check-in via Devices → Management Commands → Send Blank Push).

5
Remove the G1 profile

Once all devices show the G3 profile as installed in Jamf inventory, edit the old G1 configuration profile and click Delete. Jamf will remove it from devices on next check-in.

✅
Confirmation: Jamf Pro inventory shows profile status as Installed for all in-scope devices. RADIUS logs show successful EAP-TLS handshakes.

💻 macOS

Mac computers — Keychain manual install and Jamf Pro deployment

Manual Installation Manual

Prerequisites

  • macOS 12 Monterey or later
  • Local administrator account
  • DigiCert G3 root CA .cer file downloaded from SecureW2 portal
  • Access to Keychain Access app or Terminal
1
Download the G3 root certificate

Log in to the SecureW2 JoinNow portal. Download the DigiCert G3 Root CA .cer file to your ~/Downloads folder.

2
Add to System keychain

Open Keychain Access (Applications → Utilities). Select System keychain in the left panel. Drag the .cer file into the certificate list, or use File → Import Items. Enter your admin password when prompted.

3
Set trust to Always Trust

Double-click the imported DigiCert G3 certificate. Expand the Trust section. Set When using this certificate to Always Trust. Close the window and enter your admin password.

4
Remove the G1 certificate

Search for "DigiCert G1" in Keychain Access. Select the certificate in the System keychain, right-click, and choose Delete.

5
Reconnect to the network

Turn Wi-Fi off and back on via the menu bar. macOS will re-authenticate via 802.1X against the new G3-signed RADIUS certificate.

ℹ️
Terminal alternative You can also run: sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain ~/Downloads/DigiCertG3Root.cer
✅
Confirmation: Wi-Fi connects automatically without a certificate trust dialog. Keychain Access shows the G3 cert as This certificate is marked as trusted for all users.

Jamf Pro Deployment MDM

Prerequisites

  • Jamf Pro 10.40 or later
  • macOS devices enrolled and MDM-managed
  • DigiCert G3 root CA .cer file uploaded to Jamf
1
Upload the G3 certificate

In Jamf Pro, go to Settings → Global Management → PKI Certificates and upload the G3 root .cer.

2
Create a macOS Configuration Profile

Go to Computers → Configuration Profiles → New. Add a Certificate payload. Select the G3 cert. Set System keychain as the target.

3
Add a Wi-Fi payload (if needed)

If your existing Wi-Fi profile references the G1 CA, update the Wi-Fi payload within the same profile to point to the G3 cert for server certificate verification.

4
Scope and deploy

Assign the profile to the appropriate computer groups. Save and push via blank MDM command.

5
Retire the G1 profile

After confirming successful deployment, delete the G1 configuration profile from Jamf Pro. It will be removed from devices on next check-in.

✅
Confirmation: Jamf inventory shows the G3 profile installed on all targets. RADIUS authentication logs show no G1-related errors.

🖥️ Windows

Windows 10 / 11 — manual certificate store and Intune deployment

Manual Installation Manual

Prerequisites

  • Windows 10 21H2 or later / Windows 11
  • Local administrator or domain admin rights
  • DigiCert G3 root CA .cer file from SecureW2 portal
1
Open the certificate store

Press Win + R, type certlm.msc, and press Enter. This opens the Local Machine certificate store.

2
Import the G3 root certificate

Expand Trusted Root Certification Authorities → Certificates. Right-click → All Tasks → Import. Follow the wizard, selecting the G3 .cer file. Place it in the Trusted Root Certification Authorities store.

3
Verify the import

Refresh the list. Confirm the DigiCert G3 root appears with a valid thumbprint matching the one published in the SecureW2 portal.

4
Remove the G1 certificate

Locate the DigiCert G1 root in the same store. Right-click → Delete. Confirm deletion.

5
Reconnect to the network

Disconnect and reconnect to the 802.1X Wi-Fi or VPN. Windows should authenticate without a certificate warning.

✅
Confirmation: Network adapter status shows Connected with no authentication errors in Event Viewer under Applications and Services Logs → Microsoft → Windows → WLAN-AutoConfig.

Microsoft Intune Deployment MDM

Prerequisites

  • Microsoft Intune tenant with Windows device management enabled
  • Devices enrolled in Intune (Hybrid AAD or AAD joined)
  • DigiCert G3 root CA .cer file
  • Intune admin role: Device Configuration Manager or higher
⚠️
Do not remove the G1 profile until all devices have the G3 profile installed Removing the G1 trusted certificate profile before a device receives the G3 profile will leave that device with no trusted CA for your RADIUS server — breaking 802.1X authentication immediately. Stage the G1 removal only after Intune reports 100% G3 profile success across all targeted devices.
1
Create a Trusted Certificate profile for G3

In Microsoft Intune admin center, go to Devices → Configuration → Create → New Policy. Select Windows 10 and later as platform and Trusted certificate as profile type.

2
Upload the G3 root certificate

On the Configuration settings page, click Upload a certificate and select the DigiCert G3 .cer file. Set Destination store to Computer certificate store – Root.

3
Assign to device groups

On the Assignments page, add the target Entra ID device groups. Use a pilot group first. Click Next and Create.

4
Monitor rollout

Go to Devices → Monitor → Assignment status for the profile. Wait until Succeeded count matches total enrolled devices in scope before proceeding.

5
Remove the G1 Trusted Certificate profile

Only after full G3 rollout: navigate to the G1 Trusted Certificate profile in Intune. Remove all assignments, then delete the profile. Intune will remove the G1 cert from devices at next sync.

✅
Confirmation: Intune shows G3 profile as Succeeded for all devices. G1 profile shows 0 assignments. No RADIUS authentication failures in your NPS or SecureW2 logs.

📲 Android

Android 10+ — Intune deployment and manual install

Microsoft Intune Deployment MDM

Prerequisites

  • Android 10 or later (Enterprise, Work Profile, or COPE enrollment)
  • Devices enrolled in Intune with Android device administrator or Android Enterprise profile
  • DigiCert G3 root CA .cer file
1
Create a Trusted Certificate profile

In Intune admin center, go to Devices → Configuration → Create → New Policy. Select Android Enterprise as platform. Choose the appropriate profile type for your enrollment (Fully Managed, Dedicated, or Corporate-Owned Work Profile) and Trusted certificate as profile type.

2
Upload the G3 certificate

Upload the DigiCert G3 root .cer file. Set destination to CA certificate.

3
Assign and deploy

Assign to your Android device groups. Save and create. Devices will receive the cert at next Intune sync.

4
Update Wi-Fi profile

If your Android Wi-Fi configuration profile specifies the CA certificate for EAP validation, edit it to reference the G3 cert instead of G1.

✅
Confirmation: Intune shows Trusted Certificate profile as Succeeded. Android devices authenticate to 802.1X Wi-Fi without manual trust prompts.

Manual Installation Manual

Prerequisites

  • Android 10 or later (unmanaged or personally-owned devices)
  • G3 root CA .cer or .crt file accessible on device (email attachment, browser download, or USB)
  • Device screen lock set (required to install CA certificates)
1
Transfer the G3 certificate to the device

Download the DigiCert G3 root .cer file from the SecureW2 portal directly on the Android device, or email it and open from your mail app.

2
Install the certificate

Go to Settings → Security → Encryption & Credentials → Install a certificate → CA Certificate. Tap the downloaded file and confirm installation. (Path may vary by manufacturer.)

3
Update the Wi-Fi network settings

Forget and rejoin the 802.1X Wi-Fi network. Under CA Certificate in the Wi-Fi advanced settings, select the newly installed DigiCert G3 root.

4
Remove the G1 certificate (optional)

Go to Settings → Security → Trusted Credentials → User and remove the DigiCert G1 root if present.

✅
Confirmation: Device connects to 802.1X Wi-Fi without a certificate warning. Under Trusted Credentials the G3 root appears in the User store.

🔗 Cisco Meraki

Updating the RADIUS server CA trust in Meraki Dashboard

Prerequisites

  • Meraki Dashboard admin access (Network Admin or higher)
  • DigiCert G3 root CA .pem or .cer file
  • Existing RADIUS server configuration in Meraki (for WPA2-Enterprise SSIDs)
⚠️
Perform during a maintenance window Updating the RADIUS CA in Meraki requires saving the SSID configuration. This may briefly disrupt active 802.1X sessions on that SSID.
1
Navigate to the SSID configuration

In Meraki Dashboard, go to Wireless → Configure → SSIDs. Click Edit settings for the WPA2-Enterprise SSID used for RADIUS authentication.

2
Open RADIUS settings

Scroll to the RADIUS section. Click Edit RADIUS servers.

3
Upload the G3 CA certificate

Under RADIUS CA Certificate, click Upload certificate. Paste the PEM-encoded G3 root CA contents or upload the .cer file. Save the RADIUS server entry.

4
Save SSID configuration

Click Save changes at the bottom of the SSID settings page. The new CA will take effect immediately.

5
Remove the G1 CA (after verification)

Once you have confirmed successful authentication with the G3-signed RADIUS cert, return to the RADIUS CA Certificate field and remove the G1 certificate entry.

✅
Confirmation: Meraki event log shows successful RADIUS authentication events. No EAP failure or RADIUS timeout errors in Network-wide → Event log.

☁️ VMware Workspace ONE

UEM console — CA upload and profile push

Prerequisites

  • Workspace ONE UEM 22.x or later
  • Organization Group admin access
  • DigiCert G3 root CA .cer / .pem file
  • Devices enrolled in UEM with applicable OS profiles
1
Upload G3 CA to UEM

In the Workspace ONE UEM console navigate to Resources → Profiles & Baselines → Certificates → Certificate Authorities → Add → Upload. Upload the G3 root .cer and name it DigiCert-G3-Root.

2
Create or update a Device Profile

Go to Resources → Profiles & Baselines → Profiles → Add → Add Profile. Select the target platform (iOS, Android, macOS, or Windows). Add a Credentials payload, select Upload as the credential source, and choose the G3 CA uploaded in step 1.

3
Update the Wi-Fi payload

If the profile includes a Wi-Fi payload for EAP settings, update the Trusted Certificates or CA Certificate reference to point to the G3 CA.

4
Assign and publish

Assign the profile to the appropriate Smart Group. Click Save & Publish. UEM will push the profile to enrolled devices.

5
Remove the G1 profile

After confirming the G3 profile is installed across all devices in the Devices view, navigate to the G1 profile and click Delete. Workspace ONE will remove it from devices.

✅
Confirmation: Workspace ONE dashboard shows the G3 profile as Installed for all targeted devices. RADIUS authentication succeeds post-migration.

🔐 Okta Device Trust

Updating the Device Trust CA in Okta Admin Console

Prerequisites

  • Okta tenant with Device Trust enabled (Okta Identity Engine or Classic with Device Trust add-on)
  • Super Admin or Org Admin role in Okta
  • DigiCert G3 root CA .pem file
  • MDM integration already configured in Okta (Jamf, Intune, Workspace ONE, or similar)
ℹ️
Device Trust and RADIUS CA are separate trust stores Okta Device Trust uses certificates to verify that a device is managed. If your RADIUS server certificate chains to DigiCert G3 and is presented during an Okta-enforced network policy, the Device Trust CA configuration must also reference G3 to complete the chain of trust.
1
Navigate to Device Trust settings

In the Okta Admin Console go to Security → Device Trust. Locate the configuration for your MDM platform integration.

2
Upload the G3 CA certificate

Under CA Certificates (or MDM CA Certificate depending on your Okta version), click Add Certificate. Paste the PEM contents of the DigiCert G3 root CA or upload the .pem file.

3
Verify the certificate chain

Click Test configuration if available, or authenticate a test device through your Okta-protected app to confirm the G3-chained certificate is accepted.

4
Update Sign-On Policies (if applicable)

If you have Okta sign-on policies or app-level policies that reference a specific CA thumbprint or certificate constraint, update them to reference the G3 certificate instead of G1.

5
Remove the G1 CA entry

Once G3 trust is confirmed and all devices are passing Device Trust evaluation, delete the G1 CA certificate entry from Okta Device Trust settings.

⚠️
Removing G1 too early will fail Device Trust evaluations Devices that have not yet received the G3 certificate via MDM will fail Okta Device Trust checks if G1 is removed from Okta before their MDM profile updates. Coordinate with your MDM rollout timeline.
✅
Confirmation: Okta System Log shows successful Device Trust evaluations with no Certificate chain validation failed errors. Users on managed devices access Okta-protected resources without interruption.
